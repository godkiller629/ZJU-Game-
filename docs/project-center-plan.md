# 项目中心开发计划

## 1. 文档目标

这份文档用于指导项目功能的后续开发，只落地方案，不修改现有核心代码。

当前目标不是把所有路线做成重剧情事件链，而是在现有属性驱动框架上，新增一个统一的项目中心，让玩家的长期目标、短期经历和保研后续流程都能在同一页面中完成。

这份方案默认服务于一个较为成熟的 demo 版本，因此强调：

- 能完整打通一局
- 能体现不同人生路线
- 能兼容现有月度循环
- 不把系统复杂度一次性拉爆

## 2. 总体设计原则

### 2.1 底层原则

- 底层仍然是属性驱动
- 所有项目、经历、资格流程，最终都要回收到属性、路线分、资格状态和结局摘要变量
- 事件只做包装和反馈，不单独决定命运

### 2.2 页面原则

- 只做一个统一的项目中心页面
- 不为 SRTP、考研、实习、就业分别做独立页面
- 页面中通过不同区块承载不同类型的内容

### 2.3 现实化原则

- 学生可以有一个长期主线目标
- 同时允许存在一个短周期经历
- 保研不是从大一开始推进的长期项目
- 保研应先获得资格，再进入联系导师阶段

## 3. 项目中心的统一结构

项目中心采用三层结构：

1. 主项目层
2. 短期经历层
3. 资格后续层

### 3.1 主项目层

主项目代表一个阶段性的长期人生目标，只允许同时存在 1 个活跃主项目。

首版建议纳入：

- SRTP
- 考研准备
- 长期实习
- 就业准备

这些内容的共同特征是：

- 周期较长
- 会跨越多个月份
- 有阶段性推进
- 会显著影响毕业路线
- 会影响结局判断

### 3.2 短期经历层

短期经历不是主线目标，而是附加履历。首版建议单独开放 1 个短期经历槽。

首版建议纳入：

- 短期实习
- 暑研
- 企业体验
- 短学期实践

这些内容的共同特征是：

- 周期短，通常 1 到 3 个月
- 更贴近现实里的履历补充
- 不应该和主项目抢同一个槽位
- 影响技能、社交、科研分、就业分等变量
- 可以成为主项目的辅助经历

### 3.3 资格后续层

资格后续层当前最重要的是保研。

保研不作为普通项目，不从大一开始推进，而采用两段式结构：

1. 资格判定阶段
2. 联系导师阶段

资格判定阶段由统一规则给出：

- GPA
- 科研成果
- 路线分
- 关键项目完成情况

当玩家拿到保研资格后，项目中心顶部才解锁联系导师流程卡。

联系导师流程卡不占主项目槽，也不占短期经历槽。

## 4. 页面信息架构

项目中心页面建议拆成五个区块。

### 4.1 顶部路线看板

展示当前人生路径的核心信息：

- 当前主路线倾向
- 科研分
- 升学分
- 就业分
- 保研资格状态
- 毕业关键条件摘要

若已获得保研资格，则在这里额外展开联系导师流程卡。

联系导师流程卡建议展示：

- 目标导师
- 联系状态
- 回复结果
- 接收意向
- 导师匹配方向

### 4.2 当前主项目工作区

这里只展示唯一活跃主项目。

内容建议包括：

- 项目名称
- 类型
- 当前阶段
- 进度条
- 剩余月份
- 里程碑
- 风险提示
- 当月可执行操作

### 4.3 短期经历区

该区块专门承载短期实习与其它短经历。

内容建议包括：

- 当前进行中的短期经历
- 可报名的短期经历
- 周期与开始时间
- 收益与代价
- 是否与主项目冲突

### 4.4 可启动项目区

这里展示当前条件下可开启的新内容。

可分为两类：

- 可启动主项目
- 可报名短期经历

每张卡片建议展示：

- 名称
- 条件
- 周期
- 预期收益
- 预期代价
- 是否已解锁

### 4.5 归档区

归档区用于保留玩家过去做过的事情。

建议记录：

- 已完成主项目
- 已放弃主项目
- 已完成短期经历
- 已失败经历
- 关键成果摘要

这个区块很重要，因为它决定玩家在结局时能否回看自己这一局到底经历了什么。

## 5. 统一项目模型

建议所有主项目和短期经历都进入同一个 projects 数组，只通过字段区分类型。

首版建议结构如下：

```js
{
	id: 'project_xxx',
	kind: 'main' | 'experience',
	templateId: 'srtp' | 'kaoyan' | 'long_internship' | 'job_prep' | 'short_internship' | 'summer_research',
	category: 'research' | 'postgraduate' | 'practice' | 'career',
	status: 'locked' | 'available' | 'active' | 'completed' | 'failed' | 'abandoned',
	phase: 'proposal' | 'progress' | 'sprint' | 'wrapup',
	progress: 0,
	quality: 0,
	monthsRemaining: 0,
	startAt: { year: 2024, month: 7 },
	endAt: { year: 2024, month: 9 },
	season: 'term' | 'summer' | 'winter' | 'any',
	milestones: [],
	rewards: {},
	routeImpact: {
		research: 0,
		postgraduate: 0,
		career: 0
	}
}
```

## 6. 保研资格后的联系导师模型

保研流程不要塞进 projects 数组里，而建议单独作为资格后续状态。

建议增加如下结构：

```js
baoyanFlow: {
	qualified: false,
	unlockedAt: null,
	targetMentorId: null,
	contactStatus: 'locked' | 'available' | 'contacted' | 'replied' | 'accepted' | 'rejected',
	replyScore: 0,
	accepted: false
}
```

设计理由：

- 保研资格是结果，不是长期项目
- 联系导师是资格后的流程，不是和 SRTP、考研争同一资源位的主项目
- 这样更符合现实，也更容易和现有 GPA 判定逻辑衔接

## 7. 各路线如何落在同一页面

### 7.1 SRTP

- 归类：主项目
- 方向：科研线
- 主要收益：学识、技能、科研分、保研辅助权重
- 页面位置：当前主项目工作区

### 7.2 考研准备

- 归类：主项目
- 方向：升学线
- 主要收益：学识、升学分
- 主要代价：精力、健康、机会成本
- 页面位置：当前主项目工作区

### 7.3 长期实习

- 归类：主项目
- 方向：就业线
- 主要收益：技能、社交、金钱、就业分
- 页面位置：当前主项目工作区

### 7.4 就业准备

- 归类：主项目
- 方向：求职线
- 主要收益：就业分、社交、技能
- 页面位置：当前主项目工作区

### 7.5 短期实习经历

- 归类：短期经历
- 方向：实践补充
- 主要收益：技能、社交、金钱、就业分
- 可能副作用：健康下降、学业推进受影响
- 页面位置：短期经历区

### 7.6 保研联系导师

- 归类：资格后续流程
- 解锁时机：获得保研资格后
- 页面位置：顶部路线看板中的资格流程卡
- 不占主项目槽
- 不占短期经历槽

## 8. 月循环中的结算关系

为了不破坏现有系统，建议项目中心仍然依附现有月推进逻辑。

每次进入新月份时，至少做以下几件事：

1. 结算主项目推进
2. 结算短期经历推进
3. 检查是否达成里程碑
4. 更新路线分
5. 检查是否满足保研资格
6. 若已获得资格，则解锁联系导师流程卡

所有结果最终仍回收到：

- 属性
- 路线分
- 保研资格
- 毕业资格
- 结局摘要变量

## 9. demo 首版建议范围

为了避免系统膨胀，首版建议严格控制在以下范围：

- 1 个主项目槽
- 1 个短期经历槽
- 1 个保研资格后联系导师流程卡
- 4 类主项目模板
- 2 到 3 类短期经历模板
- 统一项目中心页面

首版不建议做：

- 完全多项目并行
- 复杂团队协作
- 多导师并行沟通系统
- 大量剧情树分支

## 10. 开发顺序建议

建议后续开发按这个顺序推进：

1. 先定状态字段
2. 再定项目模板
3. 再做项目中心页面骨架
4. 再接主项目推进逻辑
5. 再接短期经历槽
6. 最后接保研资格后的联系导师流程

这样做的好处是，保研联系导师不会反过来绑架前面所有系统。

## 11. 当前明确结论

- 保研不是大一开始推进的长期项目
- 保研应采用资格判定后再联系导师的现实流程
- 短期实习经历应正式进入项目中心
- 项目中心应统一承载主项目、短经历和资格后续流程
- 所有内容最终仍然回收到属性和统一结局系统
